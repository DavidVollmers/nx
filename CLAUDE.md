# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Repository overview

This is the Nx monorepo for `@dev-tales/nx-python`, an Nx plugin that generates and manages Python
libraries (built on [uv](https://docs.astral.sh/uv/)) inside an Nx workspace. The repo itself is both
the plugin's source (`plugins/python`) and a workspace that exercises the plugin via a test project
(`test-projects/python-lib`).

- `plugins/python` — the `@dev-tales/nx-python` Nx plugin (TypeScript, compiled with `@nx/js:tsc`).
- `test-projects/python-lib` — a Python library generated by the plugin, used to manually/e2e verify
  generators and executors against a real `pyproject.toml` / uv workspace.
- Root `pyproject.toml` is the uv workspace root; `test-projects/python-lib` is a uv workspace member.

## Commands

All tasks run through Nx (`nx` is a devDependency, invoke it with `npx nx ...` or your global `nx`).

```bash
nx build python            # build the plugin (plugins/python -> dist/plugins/python)
nx test python              # run vitest for the plugin
nx lint python               # eslint for the plugin
nx test python --coverage   # coverage report -> coverage/plugins/python

nx run-many -t build         # build all projects
nx run-many -t test          # test all projects
nx run-many -t lint          # lint all projects
```

Run a single test file with vitest directly (uses `plugins/python/vite.config.mts`):

```bash
npx vitest run plugins/python/src/generators/lib/lib.spec.ts
```

Targets for `python`-typed projects (e.g. `python-lib`, generated dynamically by the plugin's
`createNodesV2` in `plugins/python/src/index.ts` from any `pyproject.toml` with a sibling
`project.json`): `lint`, `test`, `build`, `publish` (see Architecture below).

Releases are handled by `nx release` (see `release` config in `nx.json`) — versioning uses
conventional commits, tags are `{version}`, and projects release independently. Do not run release
commands unless explicitly asked; it mutates changelogs, versions, and git tags/commits.

Formatting is prettier via `lint-staged` (`*.*": "prettier --write"`), wired up through Husky
(`pre-commit` runs `lint-staged`, `commit-msg` runs commitlint with `@commitlint/config-nx-scopes`,
which derives valid commit scopes from Nx project names). Commit messages must follow Conventional
Commits with a valid Nx project/workspace scope.

## Architecture

### Plugin registration (`plugins/python/src/index.ts`)

The plugin is registered in `nx.json` (`"plugins": ["@dev-tales/nx-python"]`) and uses Nx's
`createNodesV2` API to infer projects from `**/pyproject.toml` files. A `pyproject.toml` only becomes
an Nx project if its directory also contains a `project.json` — otherwise it's ignored (this lets
plain uv workspace members that aren't Nx projects coexist). For each qualifying directory it
synthesizes four targets (`lint`, `test`, `build`, `publish`), each backed by a custom executor named
by the constant `PLUGIN_NAME` (`@dev-tales/nx-python`) from `plugins/python/src/constants.ts`.

### Generators (`plugins/python/src/generators/`)

- `init` — idempotently registers the plugin in `nx.json`, scaffolds a root `pyproject.toml` from
  `files/root/__tmp__pyproject.toml` (Nx template, `KeepExisting` strategy), and extends
  `.gitignore` with Python-specific entries. Called automatically by `lib`.
- `lib` — the main generator. Runs `init` first, resolves the project name/root via
  `determineProjectNameAndRootOptions` (from `utils/project-name-and-root-utils.ts`), templates the
  library files (`files/lib` + optionally `files/<unitTestRunner>` e.g. `files/pytest`), and updates
  the root `pyproject.toml` to register the new package as a workspace member/dependency (via
  `utils/toml.ts`'s `updateToml`). It then queues post-generation tasks (add linter/test-runner as
  uv dev dependencies, `uv sync`) that run via `runTasksInSerial` — these are skipped when
  `--dryRun`/`-d` is passed (see the `TODO` about dry-run handling referencing an open Nx issue).

Nx template files use the `__tmp__` / `__module__` naming convention and `.template` substitution
handled by `generateFiles`; `tmp` and `module` are always passed as substitution variables even when
unused, per existing generator calls.

### Executors (`plugins/python/src/executors/`)

All executors shell out via `child_process.execSync`, using `DEFAULT_EXEC_OPTIONS` from
`constants.ts` (inherits stdio unless `NX_GENERATE_QUIET=true`, `windowsHide: false` for
Windows compatibility).

- `build` / `publish` go through `utils/uv.ts`'s `uvExecutor`, which runs `uv <command>` in the
  project root and forwards `args.__unparsed__` as additional CLI args. `build` targets
  `dist/<projectRoot>`; `publish` publishes the `*.whl` from that same output dir.
- `test` goes through `utils/dependencies.ts`'s `dependencyExecutor`, which maps required Python
  packages (e.g. `pytest`) to the shell commands that run them.
- `lint` uses `utils/linter.ts` similarly for the configured linter (currently flake8).

### Key utilities (`plugins/python/src/utils/`)

- `uv.ts` — `sync()` (`uv sync --all-packages --all-groups`), `addDependency()` (`uv add ... --group
<group> --no-sync`), and the generic `uvExecutor`.
- `toml.ts` — read/update `pyproject.toml` via `smol-toml`, used to keep the uv workspace
  (`[tool.uv.workspace]`, `[tool.uv.sources]`) and `[project.dependencies]` in sync with Nx projects.
- `linter.ts` / `unit-test-runner.ts` — normalize/prompt for the `linter` and `unitTestRunner`
  generator options (`generator-prompts.ts` backs the interactive prompts).
- `gitignore.ts` — idempotent `.gitignore` block extension.

### Release integration

`plugins/python/src/release/version-actions.ts` exists to plug into Nx's custom release
version-actions API, but is currently _not_ wired up in `index.ts` (commented out) pending
[nrwl/nx#34272](https://github.com/nrwl/nx/issues/34272) landing custom version actions support for
inferred/plugin projects.

## Conventions

- The workspace targets Python `>=3.13` (root `pyproject.toml`) and TypeScript is compiled with
  `@nx/js:tsc` targeting `es2015`/`esnext` (see `tsconfig.base.json`); the plugin package itself is
  published as CommonJS (`plugins/python/package.json` `"type": "commonjs"`).
- Module boundaries are enforced via `@nx/enforce-module-boundaries` in `eslint.config.mjs` — every
  project currently has an open `sourceTag: '*'` / `onlyDependOnLibsWithTags: ['*']` constraint.
- `.cursor/rules/nx-rules.mdc` and `.github/instructions/nx.instructions.md` are gitignored — they
  are Nx-generated AI editor rules (via `nx sync`/IDE integration), not checked in.
